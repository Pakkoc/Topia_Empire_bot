## 2. 문서 구조 및 포맷팅

모든 섹션은 **인용구(`>`)**를 사용하여 그룹화합니다.

### A. 메인 타이틀
```markdown
# <이모지> {기능 이름}
# <이모지> {기능 이름} <이모지>  (장식용 이모지가 있을 경우 양옆 배치)
```

### B. 단계/섹션 구분
```markdown
> ## {N}단계: {단계명}
> ## <이모지> {섹션명}
```

### C. 세부 내용 및 리스트
```markdown
> * **{강조 항목}:** {내용}
> - {일반 항목}
> <a:DH_MINA_03:1462813029577130200> {상세 설명 또는 하위 항목}
> <a:DH_MINA_03:1462813029577130200> {화살표 역할}
```

### D. 코드블럭 활용 (안내문, 로그, 설정값)
```markdown
> ### {소제목}
> ```{내용}
> {내용}
> {내용}```
```
*활용 예시:*
- 시스템 메시지 / 경고 문구
- 로그 예시 (날짜, 사용자, 내용)
- 권한 설정 내용
- 버튼 동작 프로세스 (1. 클릭 -> 2. 이동 ...)

## 3. 작성 원칙 (Source of Truth)

**가장 중요한 규칙입니다.** 문서를 작성할 때 기존 기획서나 문서(`docs/`)보다 **실제 코드**를 최우선으로 분석해야 합니다.

1.  **Code First**: 반드시 `apps/`, `packages/` 내의 실제 소스 코드(TypeScript)를 읽고 구현된 로직을 파악하세요.
2.  **Ignore Old Docs**: `docs/` 폴더의 내용은 기획 단계의 문서일 수 있으므로, 코드와 다를 경우 **코드의 로직**을 따릅니다.
3.  **Fact Check**:
    *   **권한**: 코드로 구현된 실제 권한 체크 로직(`isAdmin`, `hasRole` 등)을 확인하세요.
    *   **수치**: 하드코딩된 제한 값(초당 5회, 30일 등)이나 설정 기본값을 코드에서 찾으세요.
    *   **메시지**: 실제 전송되는 메시지 템플릿과 버튼 구성요소를 코드에서 확인하세요.

## 4. 톤앤매너
- **명확성**: "~함", "~임", "~기능" 등의 명사형 종결이나 간결한 서술어를 사용합니다.
- **가독성**: 긴 줄글보다는 리스트와 코드블럭을 적극 활용하여 끊어 읽기 좋게 만듭니다.
- **전문성**: "봇", "서버", "채널", "권한", "로그" 등 디스코드 전문 용어를 정확히 사용합니다.
